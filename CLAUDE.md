## Running the Bot

```bash
# Install dependencies
pip install -r requirements.txt

# Run locally (requires .env file)
python app.py

# Run with Docker (recommended for production)
docker-compose up --build
```

The bot requires a `.env` file with:
- `BOT_TOKEN` — Telegram bot token
- `ADMINS` — comma-separated Telegram IDs for super-admins
- `OPENAI_API_KEY` — used for content generation
- `ip` — host IP address
- `PRESENTON_URL` — URL to the self-hosted Presenton service (default: `http://presenton:80`)
- `PIXABAY_API_KEY` — optional, for image embedding in PPTX

## Architecture Overview

This is an **aiogram 2.x** Telegram bot that generates AI-powered presentations (PPTX) and academic documents (DOCX/PDF) for users.

### Entry Point & Startup Flow

`app.py` is the entry point. On startup it:
1. Creates all SQLite database tables
2. Starts `PresentationWorker` as a background asyncio task
3. Starts an aiohttp HTTP server on port 8080 (for frontend integration)
4. Attaches `SubscriptionMiddleware` to enforce channel subscription gating

`loader.py` wires together the bot, dispatcher, and four SQLite databases (`user.db`, `group.db`, `channel.db`, `cache.db`), all stored in the `data/` directory.

### Task Queue Pattern

The bot uses a **polling task queue** via SQLite, not an in-memory queue:
1. A user request creates a row in the `PresentationTasks` table with status `pending`
2. `PresentationWorker._process_queue()` polls every 5 seconds, picks up `pending` tasks, and processes them concurrently via `asyncio.gather`
3. On completion or failure, the task status is updated and the user is notified via Telegram message
4. On failure, any charged balance is automatically refunded

### Presentation Generation Pipeline

Two generation paths for PPTX:

**Primary** — `ProPPTXGenerator` (`utils/pptx_generator.py`):
- Built on `python-pptx`, generates fully styled PPTX locally
- 11 color themes defined in `THEMES` dict
- 7 slide layouts (title, standard, card, accent-bar, split, image+text, conclusion)
- Optionally fetches images from Pixabay API

**Fallback** — `PresentonAPI` (`utils/presenton_api.py`):
- Calls the self-hosted [Presenton](https://github.com/presenton/presenton) Docker service (`http://presenton:80`)
- Maps Gamma theme IDs → Presenton template names via `TEMPLATE_MAPPING`
- Polls the generation status endpoint until complete, then downloads the PPTX

Content for both paths is generated by `ContentGenerator` (`utils/content_generator.py`) using `gpt-4o-mini` for regular presentations and `gpt-4` for pitch decks.

### Document (Course Work) Pipeline

`CourseWorkGenerator` + `DocxGenerator` handle academic documents:
- Content is generated via OpenAI
- Output is DOCX or PDF (DOCX → PDF via LibreOffice `soffice` CLI inside the container)
- Task type is `course_work` in the `PresentationTasks` table; all other tasks are treated as presentation types

### HTTP API (Frontend Integration)

`app.py` exposes a POST endpoint at `/api/submit-presentation` (port 8080, mapped to 8081 in Docker). The frontend can submit pre-generated slide content (`pre_generated: true`) along with payment info; the bot handles balance deduction and queues the task. Auth is via `Authorization: Bearer <API_SECRET>` header.

### Database Layer

All DB classes extend `Database` (`utils/db_api/database.py`), which opens a new SQLite connection per query (no connection pooling). Key databases:
- `UserDatabase` (`users.py`) — users, balances, transactions, pricing, presentation tasks, subscriptions
- `ChannelDatabase` (`channels.py`) — mandatory subscription channels
- `GroupDatabase` / `MediaCacheDatabase` — secondary databases

### Handlers

- `handlers/users/user_handlers.py` — main user flow: FSM states for presentation creation (topic → details → slide count → theme → confirm), pitch deck Q&A (10 questions), and balance top-up
- `handlers/users/admin_panel.py` — admin commands: user management, pricing, transactions, balance adjustments, subscription plans
- `handlers/users/course_work_handler.py` — academic document creation flow

### Middleware

`SubscriptionMiddleware` (`middlewares/checksub.py`) runs before every update. It blocks all non-admin users from any action until they subscribe to all channels stored in `channel.db`. Admins (IDs in `ADMINS` env var) bypass this check.

### Themes

Theme IDs used in the frontend/keyboard are Gamma theme IDs (e.g., `chisel`, `coal`, `blues`). `utils/themes_data.py` holds the display metadata. `PresentonAPI` maps these to Presenton template names; `ProPPTXGenerator` uses its own internal `THEMES` dict keyed by different names — keep both in sync when adding themes.

### FSM State Convention

All multi-step user flows use aiogram FSM (`StatesGroup`). State classes live at the top of each handler file. The `_kb(msg)` helper in `user_handlers.py` is the standard way to build the main menu keyboard with the user's `telegram_id` injected.

## Redesign Plan (Q2 2026 — agreed direction)

The user (Staff/Principal Engineer) has signed off on a full platform rewrite. The current bot will be migrated into a marketplace platform with three revenue streams:

1. **AI Generation** (pay-per-use) — presentations, mustaqil ish, referat, kurs ishi, diplom, magistr, tezis, ilmiy maqola, krossvord
2. **Template Marketplace** — admin uploads `.pptx` templates → users buy → AI fills with their topic
3. **Ready Works Marketplace** — pre-written academic works (mustaqil ish, diplom, etc.) sold as instant downloads

### Target architecture

- **Web**: Next.js 14 App Router + TypeScript + shadcn/ui + Tailwind (new repo `aislidebot-web/`)
- **Backend API**: FastAPI + Python 3.12 + SQLAlchemy 2.0 + Alembic
- **Database**: PostgreSQL (Neon serverless) — replaces all 4 SQLite files
- **Queue**: Dramatiq + Redis (Upstash) — replaces in-process asyncio polling
- **Storage**: Cloudflare R2 — replaces local `/tmp` files
- **Auth**: Telegram Login Widget + JWT (NextAuth on web)
- **Payments**: Click + Payme (UZ providers)
- **Bot role**: notifications + file delivery only (aiogram 3.x rewrite)

### Key engineering decisions

**PPTX generation — switch to template injection.** Drop the from-scratch `ProPPTXGenerator` and Presenton fallback in favor of a `TemplateInjector` (`python-pptx` based) that fills placeholders in admin-uploaded `.pptx` templates. This preserves all original design (fonts, colors, masters, animations) — quality unattainable by code-generated layouts. Templates parsed at upload time → `layouts JSONB` column stores placeholder map. AI content generation becomes layout-aware.

**Slide-count adjustment.** When user-requested slide count differs from template, `copy.deepcopy` middle content slides to expand or trim them to shrink — preserve title and conclusion slides.

**Crossword generator (new product).** AI generates 15-30 word/clue pairs → constraint-solver places them on a grid → `reportlab` exports PDF + PNG preview.

**PDF conversion.** Replace fragile `soffice` subprocess with Playwright/Chromium HTML→PDF for academic docs (more reliable, no LibreOffice dependency).

**FTS for marketplace.** PostgreSQL `tsvector` + GIN index on `ready_works(title, abstract, subject)` — no Elasticsearch needed.

Old `aislide_frontend` is **never touched**. New `aislidebot-web` deploys to a separate Vercel project and domain. Cutover happens by updating this bot's `WebAppInfo` URLs only. AIDA bot continues using the old shared frontend until/unless its team migrates separately.

For data: `scripts/migrate_sqlite_to_pg.py` will copy users (preserving `telegram_id` + balances), transactions, and completed task references. In-flight pending tasks are refunded, not migrated.
